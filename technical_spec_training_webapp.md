# Техническое задание на разработку web-приложения для учета, контроля и анализа тренировок

## 1. Общие сведения

### 1.1. Наименование проекта
Многопользовательское web-приложение для учета, контроля, планирования и анализа тренировок.

### 1.2. Цель проекта
Разработать русскоязычное web-приложение с мобильным акцентом для использования в спортзале, предназначенное для ведения журнала тренировок, назначения тренировочных планов клиентам, хранения истории занятий, анализа прогресса и взаимодействия между ролями «тренер» и «клиент».[cite:28][cite:32]

### 1.3. Назначение системы
Система должна обеспечивать:
- учет персональных и назначенных тренировок;
- хранение структуры тренировок по датам, типам и залам;
- фиксацию фактических результатов выполнения упражнений;
- анализ прогресса по ключевым метрикам;
- разграничение прав доступа по ролям и административному признаку.[cite:32][cite:38]

### 1.4. Формат использования
Приложение должно работать как web-приложение с адаптацией под смартфон и рассматриваться как mobile-first решение. Для данного класса приложений допустимо использование PWA-подхода с поддержкой ключевых offline-сценариев и последующей синхронизации данных.[cite:28][cite:30]

## 2. Пользователи и роли

### 2.1. Базовые роли
В системе должны быть предусмотрены две основные роли:
- тренер;
- клиент.[cite:32]

### 2.2. Административный признак
Пользователь любой из базовых ролей может дополнительно обладать административным признаком. Администратор имеет право создавать учетные записи новых пользователей. Такой подход соответствует практике разграничения прав через набор разрешений, а не только через название роли.[cite:32][cite:38]

### 2.3. Связи между пользователями
Система должна поддерживать следующие связи:
- один тренер может вести много клиентов;
- один клиент может быть привязан к нескольким тренерам.[cite:32]

## 3. Основные сущности данных

Система должна оперировать следующими сущностями:
- пользователь;
- роль пользователя;
- административный признак;
- профиль тренера;
- профиль клиента;
- связь «тренер-клиент»;
- спортивный зал;
- шаблон тренировки;
- план тренировки;
- тренировка;
- упражнение в тренировке;
- подход;
- медиа-вложение.

### 3.1. Сущность «Спортивный зал»
Для каждой тренировки должна быть доступна привязка к конкретному спортивному залу. Это необходимо для последующего анализа истории занятий по местам проведения тренировок и сопоставления результатов с условиями конкретного зала.

### 3.2. Сущность «Тренировка»
Карточка тренировки должна включать как минимум следующие поля:
- дата тренировки;
- тип тренировки;
- спортивный зал;
- клиент;
- тренер (если тренировка назначена или сопровождается тренером);
- список упражнений;
- заметка к тренировке;
- самочувствие;
- вложенные медиафайлы.

### 3.3. Сущность «Упражнение в тренировке»
Для каждого упражнения в составе тренировки должны храниться:
- наименование упражнения;
- порядок упражнения в тренировке;
- список подходов;
- заметка к упражнению при необходимости.

### 3.4. Сущность «Подход»
Для каждого подхода должны храниться:
- вес;
- количество повторений;
- RPE;
- время отдыха.

### 3.5. Сущность «Медиа-вложение»
Система должна поддерживать прикрепление к тренировке или упражнению:
- фото оборудования;
- фото, снятого пользователем;
- видео, снятого пользователем.

Хранимые медиафайлы допускаются в виде сжатых копий без обязательного сохранения оригиналов. Для медиа-загрузки в ТЗ необходимо фиксировать форматы, размеры файлов и правила доступа к вложениям.[cite:33]

## 4. Функциональные требования

### 4.1. Авторизация
Система должна обеспечивать авторизацию пользователей с проверкой подлинности учетных данных. Для web/PWA-приложений безопасность авторизации и управление сессией являются обязательной частью требований, особенно при работе с персональными данными и доступом с мобильных устройств.[cite:27][cite:31][cite:40]

Минимальные требования:
- вход по логину или email и паролю;
- выход из системы;
- смена пароля;
- восстановление доступа по решению, выбранному на этапе проектирования.

### 4.2. Управление пользователями
Администратор должен иметь возможность:
- создавать пользователей;
- назначать базовую роль пользователя;
- устанавливать или снимать административный признак;
- редактировать базовые данные пользователя;
- деактивировать пользователя.

Самостоятельная регистрация пользователей не требуется.

### 4.3. Управление связями тренер-клиент
Система должна позволять:
- назначать клиенту одного или нескольких тренеров;
- привязывать тренеру множество клиентов;
- просматривать список связанных пользователей;
- ограничивать доступ к данным только пользователями, связанными между собой в рамках бизнес-логики системы.[cite:32][cite:38]

### 4.4. Ведение справочника спортивных залов
Система должна позволять:
- создавать записи спортивных залов;
- редактировать наименование зала;
- выбирать зал при создании или редактировании тренировки;
- использовать данные по залу в фильтрации и аналитике.

### 4.5. Работа с шаблонами и планами тренировок
Тренер должен иметь возможность:
- создавать шаблоны тренировок;
- назначать шаблон конкретному клиенту;
- формировать план тренировок на основе шаблонов;
- редактировать назначенные шаблоны;
- использовать шаблон как основу для фактической тренировки.[cite:1]

### 4.6. Ведение фактических тренировок
Пользователь должен иметь возможность:
- создавать новую тренировку;
- выбирать дату, тип тренировки и зал;
- добавлять упражнения;
- добавлять подходы к упражнениям;
- указывать вес, повторы, RPE и время отдыха;
- вносить заметки и оценку самочувствия;
- прикреплять фото и видео;
- редактировать сохраненную тренировку;
- удалять тренировку.

### 4.7. Импорт старых тренировок
Система должна поддерживать импорт старых тренировок из текстового формата. Импорт должен преобразовывать неструктурированный текст тренировочного журнала в структурированные сущности: тренировка, упражнения, подходы и дополнительные заметки. Данное требование напрямую связано с уже существующим пользовательским сценарием импорта текстовых записей в прототипах приложения.[cite:21][cite:23][cite:25]

### 4.8. Просмотр истории
Система должна позволять:
- просматривать историю тренировок по пользователю;
- фильтровать историю по периоду;
- фильтровать по типу тренировки;
- фильтровать по спортивному залу;
- фильтровать по упражнению;
- открывать карточку отдельной тренировки.

### 4.9. Аналитика
Обязательные аналитические функции:
- анализ прогресса по рабочим весам;
- анализ тоннажа;
- анализ частоты тренировок;
- графики по упражнениям;
- сравнение периодов.

Дополнительно рекомендуется предусмотреть:
- динамику по конкретному упражнению;
- сравнение результатов по спортивным залам;
- просмотр последних результатов по упражнению перед новой тренировкой;
- базовые KPI на главном экране.

Для web-приложений с аналитикой данные обычно представляются через KPI, графики и детализацию по периодам, а в мобильном сценарии ключевые показатели должны быть доступны без сложной навигации.[cite:28]

### 4.10. Заметки
Отдельный встроенный чат между тренером и клиентом не требуется. Вместо этого система должна поддерживать заметки внутри записи тренировки и, при необходимости, заметки на уровне упражнения.

### 4.11. Уведомления
Push-уведомления на первом этапе разработки не требуются.

## 5. Нефункциональные требования

### 5.1. Интерфейс и UX
Приложение должно разрабатываться по mobile-first принципу, так как основное использование предполагается со смартфона в спортзале. Для мобильных web-интерфейсов рекомендуется проектировать крупные зоны нажатия, минимальное число действий для ввода данных и удобную работу одной рукой.[cite:28]

Требования к UX:
- быстрый ввод тренировки с телефона;
- крупные кнопки и элементы выбора;
- минимизация числа экранов и кликов при добавлении подходов;
- сохранение удобства работы на десктопе.

### 5.2. Язык интерфейса
Интерфейс приложения должен быть реализован только на русском языке.

### 5.3. Адаптивность
Приложение должно корректно работать на смартфонах и десктопных устройствах. Основной сценарий использования — смартфон; десктоп используется для расширенного просмотра, настройки и аналитики.[cite:28][cite:30]

### 5.4. Безопасность
Система должна быть развернута по HTTPS. Для PWA и web-приложений HTTPS является обязательной основой для безопасной авторизации, service worker и защищенной передачи данных.[cite:27][cite:28][cite:31]

Также должны быть обеспечены:
- разграничение доступа по ролям и правам;
- запрет доступа к чужим данным без соответствующей связи и разрешений;
- защита медиа-вложений от несанкционированного просмотра;
- защита серверной части от неавторизованных запросов.[cite:27][cite:31][cite:38][cite:40]

### 5.5. Offline- и online-сценарии
Поскольку приложение планируется как web-решение с мобильным акцентом, необходимо предусмотреть работу ключевых пользовательских сценариев при нестабильном интернете. Для PWA-подхода стандартной практикой является кэширование интерфейса и поддержка offline-доступности основных экранов.[cite:28][cite:30]

Минимально желательно предусмотреть:
- открытие приложения при отсутствии сети;
- просмотр ранее загруженных данных;
- сохранение черновика или введенных данных с последующей синхронизацией.

### 5.6. Производительность
Система должна обеспечивать быстрый отклик интерфейса при добавлении упражнений, подходов и медиа-вложений. На мобильном устройстве основные действия пользователя не должны требовать длительного ожидания между шагами.

## 6. Права доступа по ролям

| Функция | Тренер | Клиент | Администратор (доп. признак) |
|---|---|---|---|
| Вход в систему | Да | Да | Да |
| Просмотр собственных тренировок | Да | Да | Да |
| Создание и редактирование собственных тренировок | Да | Да | Да |
| Просмотр данных связанного клиента | Да | Нет | По бизнес-правилам |
| Назначение шаблонов клиенту | Да | Нет | Только если пользователь также тренер |
| Создание пользователей | Нет | Нет | Да |
| Назначение ролей и admin-признака | Нет | Нет | Да |
| Работа со справочником залов | Да | Да | Да |

## 7. Предварительный состав экранов

В состав первой версии системы рекомендуется включить следующие экраны:
- экран входа;
- главный экран/дашборд;
- список тренировок;
- карточка тренировки;
- экран создания и редактирования тренировки;
- экран шаблонов тренировок;
- экран назначения шаблона клиенту;
- список клиентов тренера;
- профиль клиента;
- экран аналитики;
- экран управления пользователями для администратора;
- экран справочника спортивных залов.

## 8. Критерии приемки

Система считается соответствующей ТЗ, если выполнены следующие условия:
- пользователь может войти в систему под своей учетной записью;
- администратор может создать нового пользователя;
- тренер может назначить шаблон тренировки клиенту;
- клиент может создать и сохранить фактическую тренировку;
- в тренировке можно указать зал;
- к тренировке можно прикрепить сжатое фото или видео;
- можно импортировать старую тренировку из текста;
- можно построить аналитику по рабочему весу, тоннажу, частоте и упражнениям;
- можно выполнить фильтрацию истории по залу, периоду и упражнениям;
- интерфейс корректно работает на смартфоне.

## 9. Рекомендации к следующему этапу

Для перехода от черновика ТЗ к разработке желательно дополнительно зафиксировать:
- способ авторизации и восстановления доступа;
- полный перечень полей профиля пользователя;
- формат импорта текстовых тренировок и правила парсинга;
- список аналитических экранов и видов графиков;
- ограничения по размеру фото и видео;
- требования к серверной архитектуре, API и базе данных;
- правила хранения и сжатия медиафайлов.

## 10. Комментарий по развитию проекта

С учетом требований многопользовательности, ролей, медиа-вложений, аналитики и мобильного использования данный проект уже следует рассматривать не как простой личный дневник, а как полноценное клиент-серверное web-приложение с ролевой моделью доступа и перспективой дальнейшего масштабирования. Для систем с разграничением прав, мобильным веб-доступом и offline-сценариями особенно важно рано зафиксировать модель авторизации, доступов и поведения при отсутствии сети.[cite:27][cite:28][cite:32][cite:38]
